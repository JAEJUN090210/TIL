# TIL - 07.12 / Network - Load Balancing

## Summary

- **Load Balancing(부하 분산)은 여러 서버에 클라이언트의 요청을 분산하여 성능, 가용성, 확장성을 향상시키는 기술**

------

## Body

- **Load Balancing(부하 분산)**
  - 하나의 서버에 요청이 집중되지 않도록 여러 서버에 트래픽을 분산
  - 서버의 과부하를 방지하고 안정적인 서비스 제공
  - 장애가 발생한 서버를 제외하고 정상 서버로 요청 전달 가능
- **동작 방식**
  1. 클라이언트가 서비스에 요청 전송
  2. Load Balancer가 요청을 수신
  3. 부하 분산 알고리즘에 따라 적절한 서버 선택
  4. 선택된 서버가 요청 처리
  5. 처리 결과를 클라이언트에게 반환
- **주요 부하 분산 알고리즘**
  - **Round Robin**
    - 서버에 요청을 순서대로 균등하게 분배
    - 가장 단순하고 많이 사용되는 방식
  - **Weighted Round Robin**
    - 서버의 성능에 따라 가중치를 부여하여 요청 분배
    - 성능이 높은 서버가 더 많은 요청 처리
  - **Least Connections**
    - 현재 연결 수가 가장 적은 서버에 요청 전달
    - 연결 시간이 긴 서비스에 적합
  - **IP Hash**
    - 클라이언트 IP를 기준으로 서버를 선택
    - 동일한 사용자가 항상 같은 서버에 연결되도록 지원(Session 유지)
- **Load Balancer의 종류**
  - **L4 Load Balancer**
    - TCP/UDP 계층에서 동작
    - IP 주소와 포트 정보를 기반으로 요청 분산
    - 처리 속도가 빠름
  - **L7 Load Balancer**
    - HTTP/HTTPS 계층에서 동작
    - URL, Host, Header 등을 기준으로 요청 분산
    - 다양한 라우팅 정책 적용 가능

------

## (+)

- 서버 과부하 방지
- 서비스의 가용성(High Availability) 향상
- 장애 발생 시 다른 서버로 요청 우회 가능
- 수평 확장(Scale-Out)이 용이
- 응답 속도 및 사용자 경험 개선
- 유지보수 시 무중단 서비스 운영 가능

------

## (-)

- Load Balancer 자체가 장애 지점(SPOF)이 될 수 있음
- 별도의 구축 및 운영 비용 발생
- 설정이 복잡할 수 있음
- 세션 유지(Session Persistence)가 필요한 경우 추가 설정 필요
- 잘못된 부하 분산 정책은 특정 서버에 트래픽이 집중될 수 있음
